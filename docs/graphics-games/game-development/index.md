# <a name="introduction-to-game-development-with-xamarin"></a>使用 Xamarin 的遊戲開發簡介

開發遊戲可以是極度令人興奮，尤其是考慮如何輕鬆進行發佈您的工作，行動平台上。 這篇文章討論的概念，並可協助您的遊戲開發與相關的技術建立遊戲，不論您的目標是建立高品質 AAA 遊戲，或只是為了好玩的程式。

本文章涵蓋下列主題：

- **遊戲與非遊戲程式設計概念**– 我們將探討一些概念，不是至遊戲開發，唯一或共用的其他類型的開發，但值得強調這裡，因為其重要性。
- **遊戲開發小組**– 這一節探討的遊戲開發人員小組內不同的角色。
- **建立的遊戲構想**– 這一節可協助您建立新的遊戲構想 – 讓新遊戲的第一個步驟。
- **遊戲開發技術**– 這裡，我們將列出一些跨平台提供的技術，可以改善您身為遊戲開發人員的產能。


## <a name="game-vs-non-game-programming-concepts"></a>遊戲與非遊戲程式設計概念

將移到遊戲開發的程式設計人員通常會遇到新的概念及開發模式。 此章節提供一些這類概念的高階檢視。


### <a name="the-game-loop"></a>遊戲迴圈

一般遊戲需要常數的移動或變更以回應使用者互動和自動的遊戲邏輯螢幕上發生狀況。 這透過功能通常稱為達成*遊戲迴圈*。 遊戲迴圈是某種類型的迴圈陳述式 （如 while 迴圈） 可以執行非常高頻率，例如 30 或 60*每秒畫面格數*。

以下是簡單的遊戲迴圈的圖表：

![](images/image1.png "這是簡單的遊戲迴圈的圖表")

我們在下面討論的技術會摘除實際 while 迴圈，但是儘管這個抽象概念的每個畫面格的更新將會出現。

程式碼的效能可能需要優先順序，即使是最簡單的遊戲。 例如： 需要 10 毫秒就能執行的函式可以有顯著的影響，遊戲 – 的效能，尤其是它會呼叫一次以上每個畫面。 如果您的遊戲執行每 30 畫面格，第二個然後表示每個畫面格必須執行底下 33 （毫秒）。 相較之下，這類函數甚至可能不明顯，如果只會執行以回應非遊戲應用程式中按一下按鈕。

邏輯，可能會執行的每個畫面格的常見類型包括：

- **讀取輸入**– 遊戲可能需要檢查是否使用者已產生互動遊戲藉由檢查輸入的硬體，例如觸控式螢幕、 鍵盤、 滑鼠或遊戲控制器。
- **移動**– amount 提供流暢的影片的視覺效果的每個畫面格，從一個位置的移動到另一個通常會移動非常小的物件。
- **衝突**– 多數遊戲需要經常測試是否重疊或交集各種物件。 我們將涵蓋在本文中稍後的章節深入的衝突。 移動和衝突可能會處理由專用的物理模擬系統。
- **檢查遊戲特定條件**– 遊戲的狀態可能受到某些情況下，例如是否播放程式獲得足夠的點，或是否配置的時間已經用完。
- **AI 行為**– 每個框架邏輯，可用來控制行為的物件不受控制播放程式，例如偵查的敵人或移動的弧形周圍的對手驅動程式。
- **轉譯**– 大多數的遊戲將會更新項目會顯示在畫面的每個畫面。 這可能會完成以回應變更造成影響遊戲 （例如層級中移動的字元） 或只是提供視覺化波蘭文 （例如下降 snow 或動畫的圖示）。

記住許多上面所列的活動可以變更的狀態，整個應用程式，而許多非遊戲應用程式通常會變更狀態，以回應所引發的事件。


### <a name="content-loading-and-unloading"></a>內容的載入和卸載作業

根據哪一種技術您使用在開發過程中可能需要以手動方式載入和卸載 （或處置） 的內容。 以手動方式載入和卸載的資產可能是基於多種原因所造成：

 - 資產可能需要很長的時間載入相對於單一框架的長度。 某些資產甚至可能需要秒鐘的時間載入，以避免在如果載入 mid 遊戲嚴重中斷的體驗。 如果載入時間是特別長的時間 （例如多個第二個或兩個） 可能會想要顯示動畫載入畫面或進度列。
 - 資產可能會耗用大量的 RAM，需要使用中管理以符合該遊戲的目標平台所提供的內容中載入了什麼。
 - 遊戲可能需要顯示多個資產超過可容納的 RAM。 「 開啟 World 」 遊戲通常會包含大型環境中的播放程式可以順暢地瀏覽 – 這是沒有載入畫面。 在此情況下，您可能需要建立自訂的系統中的資料流內容和管理的記憶體使用量。

自訂檔案格式可能需要在載入期間，處理需要自訂載入程式碼。

### <a name="content-pipelines"></a>內容管線

詞彙*內容管線*檔案會從其格式撰寫 （例如.png 影像檔案） 時的程序是指在遊戲中使用時，它最終的格式。 結束的格式而定在其上的內容類型正在使用以及其技術用來呈現內容。

部分內容的管線可能是非常快速，而且需要任何手動工作。 例如，大多數的遊戲引擎和 Api 可以載入其未處理的格式的.png 檔案格式。 相反地，更複雜的格式 （例如 3D 模型） 可能需要進行處理之前先載入，以不同的格式，而且這項處理可能需要一些時間，取決於資產的大小和複雜度。


## <a name="game-development-teams"></a>遊戲開發團隊

遊戲開發程序中的個人版中引進新的角色和項目。 大多數的遊戲開發人員不能滿足一組廣泛的技能，才能發行完整的遊戲，所以幾個專業領域的存在。 請記住，這並非完整的開發 – 只是某些較常見的問題領域清單。

- **程式設計人員**– 大多數的人閱讀這篇文章會歸類到此類別。 遊戲開發人員的程式設計人員的角色是類似於非遊戲應用程式中的程式設計人員的角色。 其職責包括撰寫邏輯來控制遊戲時，開發系統的一般工作的內容中指定的專案，新增和顯示內容，以及 – 當然 – 修正 bug 的流程。
- **2D artist** – 2D 演出者負責建立*2D 資產*。 這些遊戲的 GUI、 物件、 環境和字元包括影像檔。 如果您正在開發的遊戲是 3D，然後 2D 演出者可能無法負責環境和字元。 您可以找到免費為在遊戲的藝術[ http://opengameart.org/ ](http://opengameart.org/) 。
- **3D artist** – 3D 的演出者負責建立*3D 資產*。 這些包括 3D 模型的環境、 字元和屬性 （furniture、 工廠和其他在意的物件）。 有些小組會區別 3D 的演出者與 3D 揮灑小組的大小而定。 您可以找到免費為在遊戲的 3D 圖案[ http://opengameart.org/ ](http://opengameart.org/) 。
- **遊戲設計工具**– 遊戲設計人員負責定義播放遊戲的方式。 這可能包括設定例如遊戲、 遊戲和播放程式如何進行此遊戲的整體目標的高階決策。 遊戲的設計工具也可以參與非常詳盡的決策，將輸入對應的動作，例如定義係數移動或層級 ups 和設計層級的配置。 請記住，詞彙*設計工具*可能會參考到遊戲的設計工具或內容而定的視覺化設計工具。
- **聽起來設計師**– 音訊設計師負責遊戲的音訊資產。 有些小組可以區別負責建立音效和 「 作曲者 」，而較小的團隊可能會有負責所有音訊單一個人的個人。


## <a name="creating-a-game-idea"></a>建立的遊戲構想

設計遊戲似乎很容易執行 – 所有的唯一需求是 「 讓一些有趣。 」 不幸的是，許多開發人員發現自己在遺失時，若要建立要啟動開發了解。

遊戲設計的領域不容易說明，而且需要改善就如同藝術的習慣或程式設計，但這一節可協助您開始邁向成功。

新的開發人員應該從小規模開始。 它可能難以抗拒的誘惑重現大型、 現代視訊遊戲，但較小的遊戲可以是較佳的學習環境，更快速的進度會讓提供充分獎勵的體驗。

許多遊戲，同時用於學習，以及商業遊戲，開始為改善或修改現有的遊戲。 產生的想法的一個方式是查看其他遊戲，靈感。 例如，您可以考慮要使用的個人遊戲和嘗試找出遊戲的相關特性，讓您很有趣的事情。 您可能瀏覽、 能充分掌握遊戲的機制，或透過劇本的進度。 別忘了時搜尋新構想，請考慮 「 回溯 」 的遊戲。

另一種技術來產生新的想法是考慮特定內容類型，例如拼圖遊戲、 策略遊戲或 platformers。 開發人員所熟悉的內容類型可能會提供不錯的起點。

改造現有的遊戲也是教育的經驗，雖然這可能會限制已完成的產品商業的可行性。 建立遊戲，甚至是一種是正確的複製程序提供寶貴的教育經驗。


## <a name="game-development-technology"></a>遊戲開發技術

使用 Xamarin.Android 和 Xamarin.iOS 的開發人員有各種不同的技術提供給他們協助遊戲開發。 本節將討論一些最受歡迎的跨平台解決方案。


### <a name="cocossharp"></a>CocosSharp

CocosSharp 是 Cocos 2D 遊戲引擎的開放原始碼、 跨平台版本。 引擎會提供對 Android、 iOS、 Mac OS X、 Windows 桌面、 Windows RT 和 Windows Phone 的存取。

CocosSharp 著重於 2D 的遊戲程式開發的簡單程式設計 API。 在行動裝置上的遊戲的成長已協助 reignite 進行 CocosSharp 嗜好和類似的商業專案的基本的可行技術的 2D 遊戲開發的熱門程度。 它可作為來源的程式碼或.dll 檔案 （這可以透過 NuGet 取得），但它不提供的視覺化編輯器;因此，任何互動 CocosSharp 引擎需要程式設計的知識。

若要開始搭配 cocossharp 使用 Tiled，請參閱我們[CocosSharp 指南](~/graphics-games/cocossharp/index.md)。

遊戲搭配 cocossharp 使用 Tiled，，建立生氣忍，而且如果您要尋找多個平台已在執行遊戲，它可以是不錯的起點：

### <a name="monogame"></a>MonoGame

MonoGame 是開放原始碼、 跨平台版本的 Microsoft 的 XNA API。 MonoGame 可用來讓適用於 iOS、 Android、 Mac OS X、 Linux、 Windows、 Windows RT、 PS4、 PSVita、 Xbox One 和交換器的遊戲。

CocosSharp，像 MonoGame 會會就技術上而言不遊戲引擎，但而遊戲開發 API。 這表示使用 MonoGame 需要直接管理遊戲物件、 手動繪圖物件，以及實作常見的物件，例如數位相機和*場景圖形*（父子式階層之間遊戲物件）。 為了協助了解此差異，請考慮 CocosSharp 為建置基礎 MonoGame。 MonoGame 一般化一些特定平台技術，例如圖形、 轉譯和音訊，而 CocosSharp 提供組織及實作遊戲邏輯的程式碼。

MonoGame 不提供標準的視覺化開發環境，以便使用 MonoGame 需要程式設計的知識。

### <a name="additional-technology"></a>其他技術

上述反白顯示的技術是只提供技術的範例。 其他值得注意的技術包括：

- **Spritekit** – Xamarin 提供支援 Apple 的 Spritekit 遊戲架構，可讓您存取所有原生 API 的功能。 Spritekit 是 Apple 所建立的技術，因為它會提供與 iOS 生態系統的其餘部分密切整合。 當然，Sprite Kit 不是跨平台讓它不能在 Android 上。 如需有關使用 Spritekit 的詳細資訊，請參閱這篇文章：  [https://blog.xamarin.com/make-games-with-xamarin.ios-and-sprite-kit/](https://blog.xamarin.com/make-games-with-xamarin.ios-and-sprite-kit/)
- **Scenekit** – Xamarin 也提供支援 Apple 的 Scenekit 架構，可簡化實作到 iOS 應用程式的 3D 圖形。 Scenekit 也是由 Apple，使其具有對整合和特定平台考量 Sprite Kit 如先前所述的技術。 如需有關 Scenekit 的詳細資訊，請參閱這篇文章： [https://blog.xamarin.com/3d-in-ios-8-with-scene-kit/](https://blog.xamarin.com/3d-in-ios-8-with-scene-kit/)
- **OpenTK –** OpenTK （這代表開啟的工具組） 可讓您以 iOS、 Apple 和 Mac 的低階 OpenGL 存取硬體。 如需有關 OpenTK 的詳細資訊，請參閱 < 在主頁面：  [http://www.opentk.com/](http://www.opentk.com/)
